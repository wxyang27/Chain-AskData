# AskData Agent 面试故事线

## 1. 一句话定位

我在数据智能部-数据架构组做数据仓库开发，面对的主要用户不是数据团队内部，而是经管中心下的战略咨询、商业分析、经营分析等业务同事。这个项目的目标，是把数仓里已经沉淀的指标口径、表结构和取数经验，封装成一个面向业务的自然语言取数 Agent，让业务同事能用日常语言提出问题，系统自动理解指标、召回表字段、生成并校验 SQL，最后返回可信的数据结果。

这个项目不是单纯的 Text2SQL Demo，而是一个以“指标口径一致”和“释放基础取数人力”为核心目标的数据智能应用。

## 2. 60 秒开场话术

我做这个项目的背景是，数据架构组长期承接经管中心各部门的取数需求，比如战略咨询、商业分析、经营分析同事会频繁问门店核销收入、支付 GMV、新客老客、渠道贡献、品项份额等经营问题。问题在于，业务同事知道自己想看什么，但不一定知道应该用哪张表、哪个字段、哪个过滤条件；不同部门对同一个指标也容易出现口径不一致，比如核销收入和支付 GMV、核销人数和核销人次、新客和老客等。

所以我们先做了指标治理：每周和业务部门开指标评审会，把核心原子指标、衍生指标、定义、数据源、SQL 口径、确认方沉淀到数据字典里，同时整理连锁数仓表地图和易混淆口径。基于这些知识资产，我再搭建 AskData Agent，把指标字典、表字段、样例 SQL 和业务规则做成可检索的知识库。在线链路上，用户在飞书里提问后，系统会先做意图识别和语义解析，再通过混合检索召回相关指标和表字段，生成查询计划和 SQL，经过安全校验、执行、结果校验与修复后，把结果回传给业务同事。

最终我希望它解决两个问题：第一，业务侧取数更快、更少歧义；第二，数仓同学不用反复处理大量基础 SQL 需求，而是把精力放到模型建设、指标治理和复杂分析上。

## 3. 面试展开版

### 3.1 业务问题

项目起点不是“我要做一个大模型应用”，而是数仓工作中真实存在的三类问题：

1. 口径不一致  
   同一类经营指标在不同部门、不同报表、不同同事之间可能叫法不同、过滤条件不同。例如支付类指标和核销类指标容易混用，人数、人次、订单数也容易混淆。

2. 取数链路长  
   业务同事提出需求后，通常需要先解释业务问题，再由数仓同学判断表、字段、时间口径、过滤条件、SQL，再回传结果。大量基础取数需求会占用数仓开发人力。

3. 知识依赖人  
   表结构、指标口径、历史 SQL、常见坑很多沉淀在文档、表格和个人经验里，新同学和业务同事很难稳定复用。

### 3.2 业务资产

为了让 Agent 的回答可信，我没有直接让大模型凭空写 SQL，而是先把数据资产整理成结构化知识：

1. 指标字典  
   沉淀原子指标和衍生指标，包括指标定义、数据源表、SQL 计算逻辑、业务线、是否确认、确认方。比如核销 GMV、核销收入、支付 GMV、支付人数、新客核销收入、私域核销 GMV 等。

2. 数仓表地图  
   将连锁数仓按 DIM、DWD、DWS、DM 分层梳理，并区分 P0/P1/P2 主题。对于业务取数，优先推荐 DM 宽表和 DWS 汇总表，只有需要明细追溯时才下钻到 DWD。

3. 易混淆口径  
   专门整理容易误用的概念，例如 `tenant_id` 和 `hospital_id`、`customer_id` 和 `crm_customer_id`、`product_id` 和 `item_product_id`，以及预约、支付、核销、履约之间的关系。

这些资产是项目的核心壁垒。大模型只是推理和生成层，真正决定可信度的是指标治理和数仓知识沉淀。

## 4. 技术链路

### 4.1 离线知识构建

离线部分负责把业务文档、指标表、schema 文件和样例 SQL 变成可被 Agent 使用的知识库：

1. 从飞书文档、飞书表格、Markdown、YAML 等来源抽取指标、表、字段、口径规则。
2. 对指标和字段做标准化，形成 metric registry、schema index 和业务规则库。
3. 建立多路索引，包括关键词索引、向量索引和重排索引。
4. 维护表关系图，让系统知道哪些表可以通过哪些主键关联。
5. 构造黄金评测集，用来验证 schema 召回、SQL 生成和安全规则。

### 4.2 在线问答链路

在线链路可以按下面这条主线讲：

用户问题 -> 短期记忆追问补全 -> 意图识别 -> 指标/字段召回 -> 能力边界注入 -> 查询计划 -> SQL 生成 -> 安全校验 -> ODPS 执行 -> 结果校验 -> 飞书卡片返回

更具体一点：

1. Memory Resolution  
   对同一会话内的短追问先做补全。系统保存最近 3 轮结构化状态，包括时间范围、指标、维度、筛选条件、TopN、模板 ID 和上一轮 SQL。用户问“那上海呢”“top3”“本月呢”时，先继承最近一轮问题，再替换新的筛选或排序条件；只有用户明确说“回到刚才第一个”“北京那个”时，才回到更早轮次。

2. Router  
   判断用户是在问数据、趋势、知识、帮助，还是闲聊。只有数据查询进入 SQL 链路。

3. Semantic Contract  
   把自然语言问题解析成结构化约束，包括时间范围、指标、维度、过滤条件、排序、TopN 等。

4. Hybrid Schema RAG  
   用关键词检索保证精确匹配，用向量检索补充语义召回，再用 rerank 做最终排序，召回相关指标、字段和表。

5. Schema Graph  
   根据召回字段补齐相关表和 join key，减少多表关联时的随机性。

6. Capability Context  
   借鉴 MCP 的工具发现思想，在生成查询计划前注入当前可用数据库和工具能力。当前只暴露 `soyoung_dw`，并明确 schema retrieval、query plan、SQL safety gate、SQL execution 都是注册能力，模型不能编造数据库或绕过执行层直接连接底层数据源。

7. Query Planner  
   先生成查询计划，明确要查什么指标、用什么事实表、按什么维度聚合、需要哪些过滤条件。推理模型输出的是可审计的 `QueryPlanCoT`，不是隐藏思考链。它会按四元组组织：`database`、`processing_objects`、`operation_instructions`、`output_target`。

8. SQL Generator  
   对高频核心指标保留模板化 SQL，对复杂问题引入 LLM SQL。SQL 生成模型只负责根据已校验的 QueryPlanCoT 和 SchemaGraph 生成 MaxCompute SQL，不决定连接哪个数据库；`database` 只作为执行路由元信息传给执行层。追问补全后的完整问题和结构化硬约束会同步进入 SQL prompt，要求城市、品项、渠道、时间窗、TopN 等条件必须落到 SQL 里。

9. Safety Gate  
   限制只读查询、禁止危险语句、检查分区条件、检查必须的业务过滤条件，避免误扫全表或生成不可控 SQL。对于城市、品项这类标准名称可能带后缀的字段，要求使用 `REGEXP` 或 `LIKE`，避免生成 `city_name = '杭州'` 后无法命中 `杭州市`。

10. Executor and Validator  
   在 MaxCompute/ODPS 上执行 SQL，并检查空结果、异常值、字段缺失、口径冲突等问题。必要时进入修复或降级回复。

11. Feedback Loop  
   将错误样例、业务反馈、修复后的 SQL 回流到评测集和知识库，形成持续迭代闭环。

### 4.3 近期工程迭代

最近我围绕目标项目的 AskData 框架做了一轮对齐和增强，重点不是堆新模块，而是把已有链路变得更受控。

1. 工程卫生  
   修复了 Windows 终端中文显示说明，补齐 `pyproject.toml` 的 pytest `pythonpath` 配置，并清理了由 shell 重定向误生成的 `DATETRUNC(CURRENT_DATE()`、`DATE_SUB(CURRENT_DATE()` 文件。

2. 轻量 MCP 思想  
   当前没有升级成标准 MCP Server，而是先实现本地 `CapabilityContext`。它在 CoT 生成前告诉模型当前可用数据库和工具能力，生成后 validator 再校验 database、字段和关系。执行请求也增加了 `database` 字段，为后续多数据库路由预留结构。

3. 推理模型 prompt 优化  
   借鉴目标项目 `cot_planning` 的提示词，把四元组字段说明和操作顺序说明合并到当前 JSON prompt 中。重点要求 `processing_objects` 覆盖指标字段、筛选字段、分组/输出字段、日期字段、`dp`、口径字段和 join key；`operation_instructions` 必须按筛选、关联、聚合/计算、排序/截断、输出的顺序写清楚。

4. SQL 生成模型 prompt 优化  
   借鉴目标项目 `sql_generation` 的设计，将 `database` 从传给 SQL 生成模型的 CoT 文本中移除。SQL 模型只生成 SQL，不决定执行路由；数据库选择由 CoT 和执行层负责。

5. 评测结果  
   用 P0 评测集前 10 条样本做了同批前后对比，执行模式设为 `disabled`，因此只评估 LLM SQL 生成、SafetyGate 通过和采纳情况，不触发真实 MaxCompute 查询。修改前 LLM SQL 采纳为 6/10，采纳率 60%；修改后为 8/10，采纳率 80%。同时保留了两个未采纳样本的错误归因：一个是 `order_by_without_limit`，一个是 `division_without_nullif`，说明后续还需要继续补 SafetyGate/Repair，而不是只靠 prompt。

6. 短期记忆第一版  
   新增进程内会话记忆，保存最近 3 轮结构化状态。第一版不做长期持久化和摘要压缩，只解决高频追问：换城市、换品项、换渠道、换排名、换时间。默认第 3 轮继承最近一轮；只有用户明确回指“刚才第一个”“北京那个”时，才回到更早轮次。这体现的是滑动窗口思想，不是把全量聊天历史塞进 prompt。

7. LLM SQL + 3 轮追问  
   追问补全后，完整问题会继续进入原有 RAG、QueryPlanCoT、SQL 生成和 SafetyGate。也就是说短期记忆不是只服务模板 SQL，而是先把“那杭州呢”补全成“最近30天杭州奇迹胶原核销收入 TOP5门店”，再让 LLM SQL 按完整问题生成 SQL。真实 `LLM_ENABLED=true` 调试中，三轮追问可以被 LLM SQL 采纳。

8. 城市和品项模糊匹配治理  
   针对 `city_name = '杭州'` 无法匹配 `杭州市` 的问题，进一步约束 SQL prompt 和 SafetyGate：点名城市、品项时要求使用 `REGEXP` 或 `LIKE`。如果 LLM 生成精确等值，静态修复器会尝试改写为模糊匹配，再重新过门禁。

## 5. Agent 怎么讲

这个项目可以讲成 Agentic Workflow，而不是完全开放式多智能体系统。它的流程是相对固定的，这样更适合数据查询这种高准确性场景；但每一步都带有 Agent 的核心能力：规划、工具调用、执行、反思和记忆。

可以这样回答：

> 我没有把 Agent 做成完全自由探索式，因为数据取数更看重稳定性和可控性。我采用的是 Plan-and-Execute + Reflection 的 Agentic Workflow。系统先把用户问题拆成查询计划，再调用 schema 检索、SQL 生成、ODPS 执行、安全校验等工具，最后根据执行结果做反思和修复。这样既保留了 Agent 的自主规划能力，又能通过固定链路控制 SQL 风险。

如果面试官追问多 Agent，可以拆成：

1. 规划智能体：理解问题，拆解指标、维度、时间、过滤条件。
2. 检索智能体：从指标字典、schema、历史 SQL 中召回上下文。
3. Coder 智能体：根据计划和上下文生成 SQL。
4. Validator 智能体：检查 SQL 安全性、口径一致性和执行结果。
5. Repair 智能体：根据错误信息修复 SQL 或给出澄清问题。

## 6. 项目亮点

1. 业务驱动，而不是模型驱动  
   核心价值来自指标治理、数仓表地图、业务规则和评测闭环。

2. 面向真实数仓场景  
   项目考虑了分层建模、指标口径、分区条件、主键关联、宽表优先、ODPS 执行、安全控制。

3. 降低幻觉  
   通过指标字典、schema RAG、Capability Context、QueryPlanCoT、模板 SQL、安全校验、执行反馈来约束大模型。

4. 可评测  
   项目有 golden case、schema recall、SQL 安全规则、LLM SQL 采纳率对比和单元测试，不是只靠主观演示。

5. 可落地到业务入口  
   飞书机器人适合业务同事日常提问，也符合部门协作场景。

## 7. 当前不足

### 7.1 后训练不足

目前主要依赖提示词、RAG 和规则约束，还没有把业务问法、标准 SQL、修复轨迹沉淀成训练数据。后续可以做三步：

1. 收集真实问题、最终 SQL、执行结果、人工修正，形成 SFT 数据。
2. 将错误修复过程整理成 preference 数据，用于训练模型偏好标准口径和安全 SQL。
3. 对核心场景做小模型微调或蒸馏，让高频取数不完全依赖大模型长上下文。

面试说法：

> 我现在把后训练作为下一阶段，因为要先保证高质量数据来源。相比直接微调，我优先搭建日志、反馈和评测闭环，先把真实业务问题和标准 SQL 收集干净，再考虑 SFT 或偏好优化。

### 7.2 MCP 不足

目前已先补了轻量 MCP 思想：在 CoT 生成前注入可用数据库和工具能力清单，生成后再由 validator 校验 database、字段和关系，执行层也会透传 database 作为后续路由键。当前默认只暴露 `soyoung_dw`，`soyoung_analysis` 暂不暴露，避免模型规划到没有权限的数据源。但它还不是标准 MCP Server，没有实现 `tools/list`、`tools/call`、resources 和 prompts 等协议能力。下一步可以把能力拆成标准 MCP Server：

1. `search_metric`：查指标定义、口径、SQL 片段。
2. `search_schema`：查表字段、字段解释、表关系。
3. `validate_sql`：检查只读、安全、分区、口径规则。
4. `execute_odps_sql`：执行 MaxCompute 查询并返回结构化结果。
5. `get_query_history`：读取用户历史问题和上下文。
6. `submit_feedback`：收集业务反馈和修正样例。

面试说法：

> MCP 的价值是把 Agent 能调用的工具标准化，后续不管是飞书机器人、Web 页面还是其他 Agent，都可以复用同一套数据工具能力。

### 7.3 记忆体系的现状与不足

目前已经完成第一版短期记忆：用内存 dict 按 `session_id` 保存最近 3 轮结构化状态，包括 `time_range`、`metrics`、`dimensions`、`filters`、`top_n`、`template_id`、`last_sql`。它先判断当前问题是否是追问，再补全成一个完整问题，最后进入原有 pipeline。因此它既能服务模板 SQL，也能服务 LLM SQL。

第一版的边界也很清楚：

1. 记忆只存在进程内，服务重启后会丢失。
2. 当前只保存最近 3 轮，没有做摘要压缩。
3. 主要覆盖筛选条件替换，还没有覆盖复杂分析意图切换。
4. 没有做长期用户偏好和组织级记忆。

后续可以分三层做：

1. 短期记忆  
   从当前 3 轮滑动窗口扩展到“滑动窗口 + 摘要压缩”，既保留近期细节，也保留更早会话里的关键分析对象。

2. 长期记忆  
   保存用户偏好、常用部门、常看指标、常用门店、历史确认口径。例如经营分析同事常看门店维度，战略咨询同事常看业务线和趋势。

3. 组织记忆  
   保存经指标评审会确认过的标准口径、废弃口径、指标别名和冲突处理规则。

面试说法：

> 我没有一开始就做复杂记忆系统，而是先做一个可控的短期结构化记忆：只保存最近 3 轮关键槽位，先解决真实问数里最高频的“那上海呢”“top3”“本月呢”。这比直接把聊天记录拼进 prompt 更稳定，也更容易评测。后续再把它升级为滑动窗口加摘要压缩，并区分用户偏好记忆和组织口径记忆。

## 8. 未来路线图

1. 第一阶段：多轮问数稳定性  
   在已有 3 轮短期记忆基础上，把终端调试能力稳定接入 Web/API 和飞书机器人，继续补充换城市、换品项、换渠道、换时间、换 TopN 等高频追问样例。

2. 第二阶段：LLM SQL 采纳率提升  
   继续强化 SQL prompt、SafetyGate 和静态修复。重点处理 `ORDER BY` 缺 `LIMIT`、占比计算缺 `NULLIF`、点名筛选未保留、城市/品项精确等值匹配等问题，让 LLM SQL 在复杂组合查询里更稳定被采纳。

3. 第三阶段：MCP 工具化  
   在现有 Capability Context 基础上，把 schema 检索、指标查询、SQL 校验、ODPS 执行封装成标准工具。短期先保持本地轻量注册表，等需要被 WorkBuddy 或其他 Agent 复用时，再升级成标准 MCP Server。

4. 第四阶段：记忆体系增强  
   从进程内 dict 升级到可持久化会话记忆，并引入摘要压缩、长期用户画像和组织级指标记忆。

5. 第五阶段：后训练闭环  
   用真实问答、标准 SQL、人工修复和执行反馈构建训练数据。

6. 第六阶段：业务运营化  
   增加权限、收藏、历史查询、导出、口径自动标注、异常预警和定时推送。

## 9. 面试追问回答

### 为什么不能直接让大模型写 SQL？

因为数仓查询的核心不是 SQL 语法，而是业务口径和表字段选择。直接生成 SQL 容易选错表、混淆指标、漏分区、漏过滤条件。我的方案是先用指标字典和 schema RAG 约束上下文，再生成 SQL，并通过安全校验和执行反馈降低错误。

### 为什么要做三路检索？

关键词检索适合精确匹配字段名、指标名和表名；向量检索适合理解业务同义表达；rerank 用于在候选结果里做最终相关性排序。三者组合能兼顾召回率和准确率。

### 为什么宽表优先？

业务同事的大部分取数问题是经营分析类聚合查询，优先用 DM 宽表和 DWS 汇总表可以减少 join 复杂度，也更贴近已经治理过的指标口径。只有需要明细排查或特殊字段时，才下钻到 DWD。

### 怎么保证指标口径一致？

一方面靠组织流程，每周和业务部门评审指标，把定义、计算逻辑、数据源和确认方沉淀进指标字典；另一方面靠系统约束，SQL 生成必须优先引用指标字典中的标准口径，并在校验阶段检查关键过滤条件和字段来源。

### 为什么 SQL 生成模型不决定 database？

因为 database 是执行路由和权限边界，不应该交给 SQL 生成模型自由决定。我的做法是让推理阶段在能力清单约束下选择 database，并由 validator 校验；SQL 生成阶段只看已校验的 QueryPlanCoT 和 SchemaGraph，负责生成 SQL。这样可以减少模型编造数据库、切换数据源或绕过执行层的风险，也为后续 MCP 化做准备。

### Prompt 优化怎么证明有效？

我没有只凭主观感觉改 prompt，而是用同一批 P0 样本做前后对比。最近一次只改推理模型和 SQL 生成模型的提示词后，在 P0 前 10 条样本上，LLM SQL 生成保持 10/10，采纳从 6/10 提升到 8/10，采纳率从 60% 提升到 80%。同时我保留失败归因，发现剩余问题主要是 `ORDER BY` 缺 `LIMIT` 和占比除法没有被 SafetyGate 认可，下一步应继续补安全规则和修复策略。

### 短期记忆怎么体现滑动窗口？

我的第一版短期记忆不是保存无限上下文，而是每个 `session_id` 只保留最近 3 轮结构化状态。普通短追问默认继承最近一轮，例如第一轮问“北京奇迹胶原 TOP5”，第二轮问“那上海呢”，第三轮问“那奇迹童颜呢”，第三轮会继承第二轮的上海，再替换品项为奇迹童颜。只有用户明确说“回到刚才第一个”“北京那个”时，系统才从窗口里选择更早轮次。这就是滑动窗口在当前项目里的体现。

### 短期记忆是否只适用于模板 SQL？

不是。短期记忆发生在 pipeline 最前面，作用是把追问补全成完整问题；补全后的问题再进入 RAG、QueryPlanCoT、模板 SQL、LLM SQL 和 SafetyGate。因此它既能让模板 SQL 改写城市/品项/时间/TopN，也能让 LLM SQL 在完整语义和硬约束下生成 SQL。

### 项目最大的难点是什么？

最大的难点不是调用大模型，而是把非结构化的业务口径转成机器可用的标准知识，并在 SQL 生成时稳定约束模型。数据应用要求结果可信，所以需要把指标治理、schema 管理、SQL 安全、执行反馈和评测体系一起做。

### 如果让你继续做，你优先做什么？

我会优先做三件事：第一，把 3 轮短期记忆稳定接入 Web/API 和飞书入口，并完善真实追问评测；第二，继续提升 LLM SQL 采纳率，尤其是复杂条件组合、TopN、占比和模糊筛选；第三，把工具调用标准化成 MCP，并沉淀真实问答和修复样例，为后训练准备高质量数据。这样项目就能从 Demo 型问答工具，逐步变成可持续迭代的数据智能产品。

## 10. 简历表述

数据智能部-数据架构组 AskData Agent 项目：面向经管中心战略咨询、商业分析、经营分析等部门的自然语言取数场景，基于连锁业务指标字典、数仓表地图和历史 SQL 构建 schema RAG 与指标知识库，实现从短期记忆追问补全、业务问题理解、指标召回、能力边界注入、QueryPlanCoT、SQL 生成、安全校验、ODPS 执行到飞书卡片返回的端到端链路。项目重点解决跨部门指标口径不一致、基础取数响应慢、数仓知识复用难等问题；通过 Capability Context 约束模型可用数据库和工具，通过 template SQL + LLM SQL 受控采纳 + SafetyGate + Result Validation + Repair/Fallback 降低 LLM SQL 风险，并用 golden case、schema recall、LLM SQL 采纳率、多轮追问样例和单元测试持续评估查询可信度。
